# Zava Agent Config — Catalog (v6.1.0)

This is the human-readable index of every plugin shipped by the `zava-agent-config` **marketplace**. v6.2.0 ships 8 plugins in three tiers, aligned to [PLATFORM.md §6.1](https://github.com/DevExpGbb/agentic-sdlc-ref/blob/main/PLATFORM.md#61-layer-a--the-sdlc-ribbon) (Layer A SDLC ribbon): one **foundation** (cross-cutting baseline), six **phase kits** (one per covered SDLC phase, from Day-0 PROVISION through OPERATE), and one **accelerator** (episodic, opt-in). Consumers compose only the plugins they need.

> 🤖 _Generated by hand for Phase A. Phase B (auto-generated GitHub Pages site) tracked separately._

## How consumers use this

### Composed pin (recommended) — pick only what you need

In any Zava service repo `apm.yml`:

```yaml
dependencies:
  apm:
    - DevExpGbb/zava-agent-config/plugins/secure-baseline#v5.0.0
    - DevExpGbb/zava-agent-config/plugins/code-kit#v5.0.0
    - DevExpGbb/zava-agent-config/plugins/review-kit#v5.0.0
    - DevExpGbb/zava-agent-config/plugins/release-kit#v5.0.0
    # operate-kit, ideate-kit not needed → not pinned
```

### Marketplace pin — get the whole catalog

```bash
apm marketplace add DevExpGbb/zava-agent-config
apm install secure-baseline@zava-agent-config code-kit@zava-agent-config
```

### Then in CI

```bash
apm install
apm audit --ci --policy ./apm-policy.yaml
```

## Consumption patterns

The marketplace is a **3-tier taxonomy** (see the README diagram). Consumption rules differ per tier:

### Foundation: `secure-baseline` is **always pinned explicitly**

Every consumer redeclares `secure-baseline` in its own `apm.yml`. Phase kits do **not** declare `secure-baseline` as a transitive dep (every phase kit ships with `dependencies.apm: []`). This is intentional.

**Why no transitive shortcut?** APM resolves transitive conflicts on a *first-wins* basis (see [`apm_resolver.py:478`](https://github.com/microsoft/apm/blob/main/src/apm_cli/deps/apm_resolver.py)). For a tool dependency that is acceptable; for a **security floor** it is not. If two phase kits transitively pulled `secure-baseline` at different refs, the resolver would pick one silently, downgrading or upgrading the secure-coding instructions without an audit trail. Forcing every consumer to declare the floor explicitly means:

- A `grep -r secure-baseline */apm.yml` proves coverage across the org.
- Renovate / Dependabot bumps are visible in the consumer's PR diff, not buried in a transitive lockfile change.
- A CI presence check (see [`apm-baseline-check.yml`](.github/workflows/apm-baseline-check.yml)) can fail the build if the floor is missing — uniform across consumers because the pin is in a uniform location.

> **Anti-pattern.** Do **not** add `secure-baseline` to any phase kit's `dependencies.apm`. The audit trail becomes ambiguous and a future kit upgrade can silently drift the security floor. PRs that introduce this should be rejected in review.

### Phase kits: pin **only** the phases your repo actually exercises

Pin `ideate-kit` only if your repo receives unstructured input that becomes issues. Pin `operate-kit` only if you wire Azure SRE Agent. Don't pin kits you don't use — `apm install` is fast but compile-time scope hygiene matters.

### Accelerators: pin **on demand**, not by default

`modernize-kit` ships in v6.1.0 with two siblings — `framework-modernizer` (Express 4→5, in v6.0.0) and `nextjs-modernizer` (Next 14→15, added in v6.1.0). It is opt-in — consumers add it when they have a planned framework migration and remove it once the migration completes; accelerators are **episodic**, not compounding. Leaving an unused accelerator pinned isn't dangerous, but it muddies the "what is this repo asking the agent to do today" signal. Future accelerators (React 17→18, Spring Boot 2→3, Java 8→17) will ship as additional sibling skills inside the same kit, so consumers pin once and select the relevant skill at invocation.

### Future direction: `peer_dependencies`

The right long-term shape is a `peer_dependencies` (or `peerDependencies`) field on phase kits that **declares** the security-floor expectation and **errors at install** if the consumer hasn't pinned a compatible `secure-baseline`. This is the npm-ecosystem precedent. Tracked in [microsoft/apm#1196](https://github.com/microsoft/apm/issues/1196).

## Policy

Org-wide policy lives at [`DevExpGbb/.github/apm-policy.yml`](https://github.com/DevExpGbb/.github/blob/main/apm-policy.yml) and is automatically inherited by every repo in the org. Consumers do not need to copy it.

## CI workflows shipped to consumers

| Workflow | What it gates |
|---|---|
| `.github/workflows/apm-audit.yml` (reusable) | Required check: `apm audit --ci`. Blocks merge on policy violation, drift, missing lockfile, hidden Unicode, or secret detection. |
| `.github/workflows/self-audit.yml` (this repo only) | Dogfood: runs `apm install` + `apm audit --ci` + `apm pack` on every push. Validates marketplace builds and complies with org policy. |

### Migrating from v4.0.0 → v5.0.0

| v4.0.0 pin | v5.0.0 equivalent |
|---|---|
| `plugins/code-kit#v4.0.0` (got architect + panel-review) | `plugins/code-kit#v5.0.0` (architect only) **plus** `plugins/review-kit#v5.0.0` (panel-review) — kit boundary now matches the SDLC ribbon phase. |

> **v4.0.0 layout note** (still applies in v5.0.0): every primitive type lives under `.apm/<type>/` inside its plugin (symmetric across `agents/`, `skills/`, `instructions/`, `commands/`, `hooks/`). v3.0.0 used a mixed layout (agents/skills at plugin root, instructions in `.apm/`) which silently dropped instructions in some install configurations — see [microsoft/apm#1161](https://github.com/microsoft/apm/issues/1161).

### Migrating from v5.0.1 → v6.0.0

v6.0.0 is **additive**: existing v5.0.1 pins (`secure-baseline`, `ideate-kit`, `code-kit`, `review-kit`, `release-kit`, `operate-kit`) keep working — bump `@v5.0.1` → `@v6.0.0` and you're done. The major bump signals the new tier (Accelerators) so consumers can audit whether to opt into `modernize-kit`.

| Add to your `apm.yml` | When |
|---|---|
| `plugins/modernize-kit#v6.0.0` | You have a planned framework migration. v6.0.0: Express 4→5. **Remove the pin once the migration completes** — accelerators are episodic, not compounding. |

### Migrating from v6.0.0 → v6.1.0

v6.1.0 is **additive within the accelerator tier**: existing v6.0.0 pins (`secure-baseline`, `ideate-kit`, `code-kit`, `review-kit`, `release-kit`, `operate-kit`, `modernize-kit`) keep working — bump `@v6.0.0` → `@v6.1.0` and you're done. The minor bump signals a new sibling skill (`nextjs-modernizer`) inside `modernize-kit`; no API changes to existing skills.

| Change | When it matters |
|---|---|
| `plugins/modernize-kit#v6.1.0` adds `nextjs-modernizer` (Next 14→15) | You are planning a Next.js 14 → 15 migration. The skill ships alongside `framework-modernizer` (Express 4→5) — both available the moment you pin `modernize-kit@v6.1.0`. Invoke by name at the agent prompt. |
