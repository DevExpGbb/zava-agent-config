Every repo in the `DevExpGbb` org inherits a single APM policy file. You don't
opt in, you don't copy it — `apm install` and `apm audit --ci` discover it
automatically through GitHub's [`.github` default-config pattern](https://docs.github.com/en/communities/setting-up-your-project-for-healthy-contributions/customizing-your-projects-default-files).

This page is **rendered from the live policy** at every site build, so what
you read here is what your CI is enforcing right now.

## Required baseline: what Zava expects in every repo

The policy mandates that every repo includes the
[`secure-baseline` plugin](/zava-agent-config/catalog/secure-baseline/) in its
`apm.yml`. That single dependency carries the org-wide secure-coding
instructions, the docs style guide, and the security-reviewer persona that
every PR reviewer (human or agent) loads.

## Allowed sources & blocked packages

`apm install` resolves a dependency by walking the policy's allow / deny lists
in order — **deny always wins**. So even a repo inside an allowed org can be
blocked if its path is on the deny list.

## "Will this work?" — common scenarios

| What you want to do | What happens |
| --- | --- |
| Add `microsoft/vscode-skills/some-skill` to `apm.yml` | ✅ Allowed — `microsoft/**` is on the allow list. |
| Add `github/awesome-copilot/plugins/azure-cloud-development` | 🚫 Blocked — explicitly on the deny list. Even though `github/**` is allowed, deny wins. |
| Add `third-party-org/cool-thing` | 🚫 Blocked — outside the allow list. <a href={POLICY_EXCEPTION_NEW_ISSUE_URL} rel="noopener">Request an exception ↗</a>. |
| Drop `secure-baseline` from `apm.yml` | 🚫 CI fails on `required-packages`. The Foundation tier is non-negotiable. |
| Pin a newer `secure-baseline` than the policy version | ✅ Allowed — `require_resolution: project-wins` means your pin wins. Useful for canarying upgrades. |
| Declare an MCP server in your repo's `apm.yml` directly (not from a package) | {policy.mcp?.self_defined === 'deny' ? '🚫 Blocked — package it as a plugin in this marketplace instead.' : policy.mcp?.self_defined === 'warn' ? '⚠️ Warning emitted on every audit. Long-term: package it as a plugin.' : '✅ Allowed.'} |
| Hand-edit a file in `.github/agents/` or `.github/instructions/` | {policy.unmanaged_files?.action === 'deny' ? '🚫 Blocked — those directories are APM-managed only. Move the content into a plugin under this marketplace.' : policy.unmanaged_files?.action === 'warn' ? '⚠️ Warning. Long-term: package it.' : '✅ Allowed.'} |
| Compile to a non-allowed target (e.g. `cursor`) | 🚫 Blocked — `compilation-target` rejects it. Allowed: <code>{(policy.compilation?.target?.allow ?? []).join(', ')}</code>. |

## Verify your repo right now

```bash
# Audit your local checkout against the live org policy
apm audit --ci

# Same check that runs on every PR — exits non-zero on violation
apm audit --ci --policy DevExpGbb/.github
```

If the command exits non-zero, the output names the offending file and the
rule id (e.g. `dependency-allowlist`, `required-packages`). Cross-reference
the rule id with the impact table above.

<Aside type="tip">
  Run <code>apm audit --ci</code> locally **before** you push. Same checks, same
  output, no waiting on a red CI run.
</Aside>

## Request an exception

If a rule is blocking work you genuinely need, file an exception request — don't
work around it (the audit catches that anyway). The platform-governance team
reviews requests in their weekly triage.

**Decision SLA.** First triage within 2 business days. Time-boxed exceptions
ship as a per-repo bypass; permanent changes ship as a policy edit and roll
out org-wide on the next audit.

**Escape hatch (one-shot).** For a single CI run that absolutely must merge
before triage:

```bash
apm install --no-policy   # bypasses install-time enforcement
```

This is logged. CI will still flag the underlying issue on the next run.
